# 后端与数据库基础架构设计总结

## 1. 项目定位

本项目后端定位为一个 **前后端分离的通用后台管理系统模板**。

当前第一版主要服务现有 React 管理后台，设计目标是：

- 结构规范
- 方便后续扩展
- 接近真实企业项目
- 不过度设计
- 暂不采用微服务架构

---

## 2. 技术栈选型

### 后端语言与框架

- Java 版本：Java 21
- 后端框架：Spring Boot 3.x
- 项目形态：单 Spring Boot 应用
- 构建结构：单 Maven module
- 架构方式：模块化单体

### 数据库

- 数据库：MySQL 8.0
- 主键策略：BIGINT 自增主键
- 删除策略：逻辑删除
- 基础时间字段：
  - create_time
  - update_time

### 数据访问层

- ORM / 数据访问框架：MyBatis-Plus
- 查询策略：
  - 简单 CRUD 使用 MyBatis-Plus
  - 复杂查询使用 XML SQL

---

## 3. 权限认证设计

### 权限框架

- 认证权限框架：Sa-Token

选择原因：

- 更适合当前通用后台模板
- 上手成本低
- 配置相对简单
- 满足第一版登录认证、角色控制菜单的需求
- 暂不需要 Spring Security 的复杂安全体系

### 第一版权限模型

第一版采用：

- 用户 -> 角色 -> 菜单

暂不实现：

- 按钮权限
- 接口权限
- 数据权限

### 菜单权限处理方式

后端根据当前登录用户的角色，返回该用户可见菜单。

前端不负责从全部菜单中自行过滤权限菜单。

---

## 4. 项目结构设计

采用 **单项目 + 包级模块化**，不使用 Maven 多 Module。

推荐包结构示例：

    src/main/java/com/example/admin
    ├── AdminApplication.java
    ├── common
    ├── framework
    └── modules
        ├── auth
        └── system
            ├── user
            ├── role
            ├── menu
            ├── dept
            ├── dict
            ├── log
            └── file

对应 Java package 示例：

    com.example.admin.modules.system.file

---

## 5. 模块内部结构

每个业务模块内部采用统一分层：

    controller
    service
    service.impl
    mapper
    entity
    dto
    vo

示例：

    modules/system/user
    ├── controller
    ├── service
    │   └── impl
    ├── mapper
    ├── entity
    ├── dto
    └── vo

约定：

- DTO：接收前端请求参数
- VO：返回给前端的数据结构
- Entity：对应数据库表结构
- Mapper：数据库访问层
- Service：业务逻辑层
- Controller：接口入口层

---

## 6. 第一版数据库表范围

第一版不是只做最小 RBAC，而是包含核心权限能力和常见后台基础能力。

### 核心权限表

    sys_user
    sys_role
    sys_menu
    sys_user_role
    sys_role_menu

### 基础能力表

    sys_dept
    sys_dict_type
    sys_dict_data
    sys_login_log
    sys_oper_log
    sys_file

---

## 7. 部门设计

部门模型确定为：

- 用户只属于一个部门
- 部门使用树形结构
- 角色暂不绑定部门

---

## 8. 菜单设计

菜单模型支持：

- 目录
- 菜单
- 外链
- 图标
- 排序
- 隐藏状态

菜单用于支撑后台侧边栏、页面路由和目录分组。

---

## 9. 字典设计

字典采用两张表：

    sys_dict_type
    sys_dict_data

设计约定：

- 字典类型用于定义一类字典，例如用户状态、性别等
- 字典数据用于定义具体字典项
- 字典项支持排序
- 第一版字典项暂不做启用/禁用状态

---

## 10. 日志设计

第一版日志分为两类：

    sys_login_log
    sys_oper_log

### 登录日志

记录：

- 登录成功
- 登录失败

### 操作日志

只记录关键写操作：

- 新增
- 修改
- 删除
- 导入
- 导出

第一版暂不记录查询日志。

---

## 11. 文件管理设计

第一版文件管理采用本地文件存储。

暂不接入：

- MinIO
- 阿里云 OSS
- 腾讯云 COS
- 其他对象存储服务

文件表记录文件元信息，例如：

- 原始文件名
- 存储文件名
- 文件路径
- 文件大小
- 文件类型
- 上传人
- 上传时间
- 所属业务模块

文件需要记录业务归属，例如：

- avatar
- import
- attachment

---

## 12. API 接口规范

### 接口前缀

统一使用：

    /api

示例：

    /api/auth/login
    /api/system/users
    /api/system/roles
    /api/system/menus

### 统一响应结构

统一返回结构：

    {
      "code": 200,
      "message": "success",
      "data": {}
    }

### 统一分页结构

分页返回结构：

    {
      "records": [],
      "total": 100,
      "page": 1,
      "pageSize": 10
    }

---

## 13. 基础接口范围

第一版基础模块需要提供标准 CRUD。

涉及模块：

- 用户
- 角色
- 菜单
- 部门
- 字典
- 文件

标准接口能力包括：

- 分页列表
- 详情
- 新增
- 修改
- 删除 / 逻辑删除
- 启用 / 禁用
- 批量删除

第一版暂不做：

- Excel 导出

---

## 14. 开发辅助组件

第一版引入以下辅助组件：

- Lombok
- Knife4j
- Spring Validation
- Hutool

用途说明：

- Lombok：减少 Java 样板代码
- Knife4j：生成接口文档
- Spring Validation：请求参数校验
- Hutool：常用工具类支持

---

## 15. 数据库脚本管理

第一版暂不使用 Flyway 或 Liquibase。

采用手动维护 SQL 脚本的方式。

建议 SQL 文件目录：

    src/main/resources/sql/schema.sql
    src/main/resources/sql/data.sql
    src/main/resources/sql/init.sql

说明：

- schema.sql：表结构
- data.sql：初始化数据
- init.sql：可选的整体初始化脚本
