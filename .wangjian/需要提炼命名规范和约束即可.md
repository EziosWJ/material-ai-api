# 后端接口与配置补充设计总结

本文档总结从上一份“后端接口设计总结”之后继续确认的内容，主要包含初始化数据、接口命名规范、字段规范、文件访问、接口鉴权、日志清空、缓存、CORS、Knife4j 等补充设计。

## 1. 初始化数据设计

### 1.1 初始化基础数据

第一版需要初始化以下数据：

```text
admin 用户
admin 超级管理员角色
系统菜单
基础字典
```

规则：

```text
初始化 admin 用户，默认密码 admin123
初始化 admin 超级管理员角色
初始化系统菜单
初始化基础字典
```

---

### 1.2 初始化部门与绑定关系

第一版需要初始化根部门，并建立 admin 用户、admin 角色、菜单之间的绑定关系。

规则：

```text
初始化根部门
admin 用户归属根部门
admin 用户绑定 admin 角色
admin 角色绑定所有菜单
```

## 4. 接口命名风格

### 4.1 REST 风格

接口继续采用 REST 风格。

但资源名不使用复数。

示例：

```text
/api/system/user/page
/api/system/user/{id}
/api/system/user/batch
/api/system/user/{id}/status
```

不使用：

```text
/api/system/users/page
/api/system/users/{id}
```

---

### 4.2 分页后缀

分页接口统一使用：

```text
/page
```

示例：

```text
/api/system/user/page
/api/system/role/page
/api/system/menu/page
/api/system/dept/page
/api/system/dict-type/page
/api/system/file/page
```

---

## 5. HTTP 方法规范

统一采用以下 HTTP 方法风格：

```text
GET：查询
POST：新增
PUT：修改
DELETE：删除
PATCH：局部修改状态
```

已确认规则：

```text
批量删除使用 DELETE /batch
分配角色 / 菜单使用 PUT
```

示例：

```text
DELETE /api/system/user/batch
PUT /api/system/user/{id}/roles
PUT /api/system/role/{id}/menus
PATCH /api/system/user/{id}/status
```

---

## 6. 请求参数风格

统一请求参数规则：

```text
GET 查询参数使用 query string
POST / PUT / PATCH 使用 JSON body
批量删除使用 JSON body：{ "ids": [] }
分页参数统一为 page、pageSize
第一版不做通用排序，按后端默认排序
```

批量删除请求示例：

```json
{
  "ids": [1, 2, 3]
}
```

分页查询示例：

```text
GET /api/system/user/page?page=1&pageSize=10&username=admin
```

---

## 7. 字段命名规范

统一命名风格：

```text
数据库字段：snake_case，例如 user_name
Java 字段：camelCase，例如 userName
JSON 字段：camelCase，例如 userName
数据库表名统一使用 sys_ 前缀
```

示例：

```text
数据库字段：create_time
Java 字段：createTime
JSON 字段：createTime
```

---

## 8. 树结构字段规范

菜单和部门都需要树结构，统一字段如下：

```text
parent_id
sort_order
children
```

已确认规则：

```text
根节点 parent_id = 0
排序字段统一为 sort_order
树形返回字段统一为 children
```

示例：

```json
{
  "id": 1,
  "parentId": 0,
  "name": "系统管理",
  "sortOrder": 1,
  "children": []
}
```

## 9. 内置数据保护规范

用户、角色、菜单、部门、字典等内置数据需要统一保护。

统一字段：

```text
is_builtin tinyint
1 = 内置
0 = 普通
```

已确认规则：

```text
内置数据禁止删除
内置数据允许修改名称
内置数据禁止修改编码 / 关键标识
```

关键标识示例：

```text
username
role_code
menu permission_code
dept_code
dict_code
```

---

## 10. 布尔类字段规范

项目中的布尔类字段统一使用 `tinyint`。

包括：

```text
visible
deleted
is_builtin
```

规则：

```text
visible:
1 = 显示
0 = 隐藏

is_builtin:
1 = 内置
0 = 普通

deleted:
0 = 正常
1 = 已删除
```

说明：

```text
deleted 字段语义与普通布尔字段相反，需要单独明确。
```

## 12. 本地文件访问路径规则

文件存储采用本地文件存储，但不直接暴露服务器磁盘路径。

### 12.1 物理路径

文件物理路径按日期分组保存。

示例：

```text
uploads/2026/05/02/xxx.png
```

---

### 12.2 文件表路径字段

文件表中：

```text
storage_path：存物理相对路径
access_url：存后端访问接口，不存磁盘路径
```

示例：

```text
storage_path = uploads/2026/05/02/xxx.png
access_url = /api/system/file/1/view
```

---

### 12.3 文件访问接口

保留两个接口：

```text
GET /api/system/file/{id}/view
GET /api/system/file/{id}/download
```

规则：

```text
view 用于预览
download 用于下载
文件访问通过后端接口转发
不直接暴露服务器磁盘路径
```

## 13. 后端接口鉴权范围

接口鉴权范围确定如下：

```text
除登录接口外，其他后台接口默认都需要登录
文件 view/download 也需要登录
第一版不校验 permission_code，只预留字段
```

说明：

```text
第一版只做登录态校验和菜单可见控制。
permission_code 字段先预留，后续可扩展按钮权限或接口权限。
```

## 14. 当前用户接口归属

当前用户信息和当前用户菜单继续归属于 auth 模块。

接口：

```text
GET /api/auth/me
GET /api/auth/menus
```

当前用户修改密码和头像归属于 system/user 模块。

接口：

```text
PUT /api/system/user/me/password
PUT /api/system/user/me/avatar
```

已确认规则：

```text
/api/auth/me 用于当前登录用户信息
/api/auth/menus 用于当前登录用户可见菜单
/api/system/user/me/password 用于当前用户修改密码
/api/system/user/me/avatar 用于当前用户修改头像
```

## 15. 日志接口设计调整

日志接口不提供单条删除和批量删除。

改为提供清空接口，仅开发阶段使用。

上线时删除或禁用清空接口。

---

### 15.1 登录日志接口

```text
GET    /api/system/login-log/page
GET    /api/system/login-log/{id}
DELETE /api/system/login-log/clear
```

---

### 15.2 操作日志接口

```text
GET    /api/system/oper-log/page
GET    /api/system/oper-log/{id}
DELETE /api/system/oper-log/clear
```

---

### 15.3 日志清空规则

已确认规则：

```text
清空日志使用物理删除
清空接口不做额外角色限制
通过配置开关控制是否启用
```

说明：

```text
开发阶段可用。
生产环境通过关闭配置或删除接口避免误用。
```

---

## 16. 日志清空配置项

配置项：

```yaml
system:
  log-clear-enabled: true
```

规则：

```text
dev 默认 true
prod 默认 false
配置为 false 时，调用清空接口返回 403
```

## 18. CORS 跨域配置

前后端分离开发需要后端统一配置 CORS。

已确认规则：

```text
后端统一配置 CORS
dev 允许 http://localhost:5173
允许 Authorization 请求头
允许 GET / POST / PUT / DELETE / PATCH
prod 按实际域名配置
```

建议 dev 配置：

```text
allowed-origins: http://localhost:5173
allowed-methods: GET,POST,PUT,DELETE,PATCH,OPTIONS
allowed-headers: Authorization,Content-Type
```
